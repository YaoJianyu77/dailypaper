### 1. Q-Hitter: A Better Token Oracle for Efficient LLM Inference via Sparse-Quantized KV Cache.

*MLSys | 2024 | Zhenyu Zhang 0015, Shiwei Liu 0003, Runjin Chen, Bhavya Kailkhura et al.*

> Q-Hitter proposes selecting KV-cache tokens by both accumulated attention and quantization friendliness so sparse KV retention works better with low-bit quantization during LLM generation.

Long-context LLM inference is limited by KV-cache memory footprint and bandwidth, so sparse retention and quantization are natural compression levers. The paper argues that a simple combination of Heavy Hitter token selection and low-bit KV quantization is suboptimal because quantization can degrade token selection accuracy, especially at 4 bits or below and in deeper layers. Q-Hitter changes the token oracle by combining accumulated attention scores with layer-specific quantization-friendliness metrics, aiming to keep tokens that matter for model behavior and are also easier to quantize. The abstract claims this addresses the compound effect of sparsification and quantization, but it is truncated and does not provide quantitative accuracy, throughput, memory, model, or hardware details. The practical value is clear for KV-cache-efficient serving, but the full paper needs to establish whether the oracle overhead and baseline comparisons justify the claimed inference gains.

If validated, the method could improve LLM serving efficiency by reducing KV-cache memory and bandwidth while avoiding accuracy loss from poorly chosen quantized tokens. A key caveat is that the abstract is truncated and does not provide quantitative results or baselines. ### 2. Proteus: Portable Runtime Optimization of GPU Kernel Execution with Just-in-Time Compilation.

*CGO | 2025 | Giorgis Georgakoudis, Konstantinos Parasyris, David Beckingsale*

> Proteus is a portable GPU-kernel JIT that specializes LLVM IR at runtime for input arguments and launch parameters across AMD and NVIDIA GPUs.

This paper targets the gap between ahead-of-time GPU compilation and runtime information that could enable stronger kernel specialization. Proteus extracts language-agnostic LLVM IR, compiles it just in time, and uses a minimally intrusive annotation interface to specialize GPU kernels for inputs and launch parameters. The abstract reports evaluation on diverse programs across AMD and NVIDIA GPUs, with end-to-end speedups over AOT compilation and an average advantage over CUDA-specific Jitify. Its systems value is in portable runtime specialization rather than LLM-specific code generation. The main missing detail is how compilation overhead, cache reuse, benchmark choice, and hardware configuration affect the reported gains.

If the evaluation holds, the work is useful for compiler/runtime co-design around portable GPU specialization, including kernel-heavy systems where runtime constants are exploitable. A key caveat is that not directly about LLM inference, serving, or LLM-driven CUDA code generation despite matching the GPU-kernel focus. Another caveat is that the abstract does not specify benchmark composition, hardware setup, or how many launches are needed to amortize JIT cost.

### 3. KPerfIR: Towards a Open and Compiler-centric Ecosystem for GPU Kernel Performance Tooling on Modern AI Workloads.

*OSDI | 2025 | Yue Guan 0003, Yuanwei Fang, Keren Zhou 0001, Corbin Robeck et al.*

> KPerfIR moves GPU kernel profiling into compiler passes, aiming to make Triton-oriented AI workload analysis more programmable, reusable, and portable.

KPerfIR targets the tooling gap between GPU compilers and profilers for modern AI workloads, where kernel behavior can depend on fine-grained intra-kernel effects. The paper proposes a multilevel compiler-centric infrastructure that implements profiling functionality as compiler passes, so analysis logic can be customized and reused inside the compiler workflow. Its concrete integration point is Triton, which makes the work directly relevant to GPU kernel generation and optimization pipelines. The abstract reports 8.2% profiling overhead, 2% relative measurement error, and qualitative actionable insight into intra-kernel optimizations. The main missing detail is the evaluation breadth: the abstract does not specify GPU models, workloads, baselines, or how representative the measured kernels are for LLM inference.

If the design generalizes, it could help kernel engineers connect profiling signals to compiler-level decisions in Triton-style GPU code generation workflows. A key caveat is that the abstract does not specify evaluation hardware, workload diversity, or baseline profilers. Another caveat is that the claimed actionable insights are not tied to explicit performance improvements in the abstract.

### 4. CuAsmRL: Optimizing GPU SASS Schedules via Deep Reinforcement Learning.

*CGO | 2025 | Guoliang He, Eiko Yoneki*

> CuAsmRL uses reinforcement learning to mutate GPU SASS instruction schedules for specialized CUDA kernels, seeking throughput gains beyond compiler -O3 output.

Specialized CUDA kernels for LLM-scale tensor workloads can still lose performance at the SASS scheduling level, where expert manual tuning often relies on trial and error. CuAsmRL frames SASS schedule optimization as an assembly game that starts from an -O3 optimized schedule and lets an RL agent iteratively mutate instruction orderings. The reward comes from measured GPU throughput, so the search is tied to actual execution rather than a purely static cost model. The abstract reports transparent speedups on existing specialized CUDA kernels of up to 26% and 9% on average. The main missing detail is the evaluation setup: GPU targets, kernel set, training cost, baseline fairness, and reproducibility are not specified in the abstract.

If the reported gains hold across kernels and GPUs, this could be a practical post-compilation optimization layer for CUDA kernel generators and compiler/runtime pipelines. A key caveat is that the abstract does not disclose hardware targets, benchmark composition, or baseline details. Another caveat is that search cost and training stability are unclear, which matters for practical compiler integration.

### 5. Tackling the Matrix Multiplication Micro-Kernel Generation with Exo.

*CGO | 2024 | Adrián Castelló 0001, Julian Bellavita, Grace Dinh, Yuka Ikarashi et al.*

> The paper presents an Exo-based workflow for generating GEMM micro-kernels that aims to match hand-written intrinsics or assembly while making target descriptions more portable.

GEMM libraries still rely on small architecture-specific micro-kernels to extract peak performance, which makes each new hardware target expensive to support. This paper frames the problem as micro-kernel generation rather than full GEMM algorithm design, using the Exo compiler to build a step-by-step path from high-level kernel structure to hardware-oriented code. The key mechanism is a concise library-based description of target instructions, which is meant to separate hardware specialization from manual kernel rewrites. The abstract claims generated kernels perform close to, or sometimes better than, manually developed intrinsics or assembly micro-kernels, but it does not provide architectures, matrix shapes, metrics, or baseline details. Its practical value is strongest for compiler-driven low-level kernel portability, while its direct relevance to LLM inference or CUDA-specific generation is not established by the abstract.

If the claim holds, the work could reduce the engineering cost of maintaining architecture-specific GEMM kernels while preserving low-level performance. A key caveat is that the abstract does not specify hardware targets, metrics, workloads, or exact performance margins. Another caveat is that the work is about GEMM micro-kernels generally, not explicitly CUDA kernels or LLM inference serving paths.

### 6. FlashAttention: Fast and Memory-Efficient Exact Attention with IO-Awareness

*arXiv preprint | 2022-05-30 | Tri Dao, Daniel Y. Fu, Stefano Ermon, Atri Rudra et al.*

> FlashAttention frames exact attention as an IO problem, using tiling and kernel fusion to reduce HBM traffic rather than changing the attention computation itself.

This paper targets the memory movement bottleneck in exact attention, which is central to efficient Transformer training and inference. Its core move is an IO-aware attention algorithm that tiles computation and fuses kernels to cut HBM traffic while preserving exact attention. The provided abstract supports the mechanism-level claim that the gain comes from reduced memory traffic, not from approximation. It also positions the method as a baseline for later FlashAttention variants and modern attention kernels. The abstract does not provide quantitative results, hardware setup, baseline comparisons, or implementation details, so those need checking in the full paper.

If the implementation details hold up, this is directly relevant to attention-kernel design and the compiler/runtime assumptions behind modern LLM inference efficiency. A key caveat is that the abstract is too compressed to assess quantitative performance claims. Another caveat is that no hardware assumptions, sequence-length regimes, or benchmark settings are provided in the supplied text.
