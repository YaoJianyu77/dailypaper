### 1. Q-Hitter: A Better Token Oracle for Efficient LLM Inference via Sparse-Quantized KV Cache.

*MLSys | 2024 | Zhenyu Zhang 0015, Shiwei Liu 0003, Runjin Chen, Bhavya Kailkhura et al.*

> This paper argues that KV-cache sparsification and low-bit quantization should be co-designed, and proposes a token-selection rule that accounts for both attention importance and quantization friendliness during LLM decoding.

The paper targets a core LLM inference bottleneck: long-context KV cache storage and bandwidth during autoregressive decoding. It argues that simply combining heavy-hitter token sparsification with low-bit KV-cache quantization is not enough, because aggressive quantization can distort heavy-hitter selection and because the most attention-important tokens are not always the best tokens to quantize. The proposed Q-Hitter rule adds a notion of quantization friendliness alongside accumulated attention scores so the retained tokens are chosen to better survive sparse low-bit KV compression. From the abstract, the claimed evidence is that this addresses the failure modes of naive post-sparsification quantization, especially in deeper layers and at 4-bit or below, with the practical value being more efficient KV-cache usage under tighter memory and bandwidth budgets. The biggest missing detail in the abstract is the actual evaluation setup and how much quality or throughput improves relative to prior heavy-hitter baselines.

If the method is sound, it would make sparse low-bit KV-cache compression more reliable for LLM serving by improving the tradeoff between memory or bandwidth reduction and model quality. A key caveat is that the abstract is underspecified on benchmarks, metrics, and quantitative gains, so the practical impact is hard to judge from the summary alone. Another caveat is that q-Hitter is described as a rule of thumb, which may mean heuristic behavior that is sensitive to model architecture or decoding regime selection details are unclear from the abstract, making reproducibility and deployment cost uncertain.

### 2. From Threads to Tiles: T2T, a Compiler for CUDA-to-NPU Translation via 2D Vectorization.

*CGO | 2026 | Shuaijiang Li, Jiacheng Zhao, Ying Liu 0055, Shuoming Zhang et al.*

> This paper presents a compiler that lifts CUDA's thread-level SIMT semantics into an explicit loop form and then maps selected loop pairs onto 2-D NPU tile instructions for cross-architecture translation.

This paper targets the growing mismatch between CUDA's fine-grained thread model and NPUs that execute coarse 2-D tile instructions. It proposes T2T, a compiler pipeline that first rewrites implicit SIMT parallelism into an explicit loop-nest representation through a Unified Parallelism Abstraction, then applies 2-D vectorization to choose loop pairs that fit NPU tile hardware. The abstract claims semantics-preserving handling of control flow and warp-level intrinsics, which matters because these are common failure points for direct CUDA retargeting. Reported results say the translated kernels reach up to 73% of native CUDA performance on an A100-equivalent comparison point and beat baseline translation methods by up to 6.9x across benchmarks. The main missing detail is how representative the NPU targets, benchmark mix, baselines, and hardware-normalization choices are, so the fairness and transferability of the reported gains still need a full read.

If the method holds up, it offers a practical compiler path for reusing CUDA kernels on tile-oriented NPUs instead of rewriting kernels manually for each accelerator ISA. A key caveat is that the headline performance numbers may depend heavily on benchmark selection and how the NPU targets are modeled. Another caveat is that correctness and performance for irregular control flow, memory behavior, and complex warp primitives may be narrower than the abstract suggests.

### 3. LLM-VeriOpt: Verification-Guided Reinforcement Learning for LLM-Based Compiler Optimization.

*CGO | 2026 | Xiangxin Fang, Jiaqin Kang, Rodrigo Rocha, Sam Ainsworth 0001 et al.*

> This paper trains a small LLM for LLVM IR peephole optimization with reinforcement learning shaped by Alive2 verification signals so correctness becomes part of the optimization objective.

This paper targets a core blocker in LLM-based compiler optimization: generated IR rewrites are often incorrect, which makes performance gains unusable. The proposed method, LLM-VeriOpt, trains Qwen-3B with GRPO and uses semantic-equivalence feedback from Alive2 as part of the reward, so the model is pushed toward optimizations that are both valid and useful. The abstract claims large gains in successful code modification, high rates of verifiably correct output, and speedups relative to unoptimized code that are comparable to LLVM's handwritten `instcombine` in many cases, with some cases reportedly exceeding it. For systems readers, the interesting point is the compiler/runtime co-design angle between LLM training and formal verification rather than plain code generation. The biggest missing detail from the abstract is the evaluation setup: benchmark coverage, reward design, verifier cost, and baseline fairness need a full read.

If the claims hold, this suggests a practical path for combining formal tools with small LLMs to produce safer low-level optimizers without relying only on larger prompted models. A key caveat is that the abstract emphasizes headline gains, but baseline fairness and benchmark representativeness are not clear from the abstract alone. Another caveat is that results may depend heavily on the specific verifier, IR task scope, and peephole-optimization setting rather than general compiler optimization workloads.

### 4. Oaken: Fast and Efficient LLM Serving with Online-Offline Hybrid KV Cache Quantization.

*ISCA | 2025 | Minsu Kim 0004, Seongmin Hong, Ryeowook Ko, Soongyu Choi et al.*

> This paper targets the KV-cache bandwidth and capacity bottleneck in batched LLM serving by combining offline-calibrated outlier thresholds with online quantization and accelerator support.

The paper is positioned around a familiar LLM serving problem: batched attention is memory-bandwidth bound, and growing KV caches make both bandwidth and capacity harder to manage. It proposes a hybrid KV-cache quantization scheme that sets outlier thresholds offline and uses them online to avoid the runtime cost of full outlier detection, aiming to keep low-bit compression without losing too much accuracy. The systems angle is a hardware-software co-design, adding custom quantization/dequantization engines and memory-management units that can plug into LLM accelerators. If the method works as claimed, the gain comes from reducing KV-cache footprint and memory traffic while avoiding the online overhead that weakens prior outlier-aware schemes. The main missing detail in the abstract is the actual evaluation evidence: the abstract fragment does not clearly state the hardware setup, baselines, accuracy impact, or end-to-end serving gains.

For LLM inference systems, this could matter if it meaningfully cuts KV-cache memory traffic and footprint while preserving model quality and avoiding expensive runtime handling of outliers. A key caveat is that the abstract is truncated, so the strength of the empirical case is unclear. Another caveat is that benefits may depend on custom accelerator support, which could limit deployability on commodity GPUs or existing serving stacks.

### 5. Proteus: Portable Runtime Optimization of GPU Kernel Execution with Just-in-Time Compilation.

*CGO | 2025 | Giorgis Georgakoudis, Konstantinos Parasyris, David Beckingsale*

> This paper presents a portable JIT path that specializes GPU kernels from LLVM IR at runtime to use argument and launch information that static compilation cannot exploit.

The paper targets a familiar systems gap in GPU HPC code: ahead-of-time compilation cannot use runtime information about inputs and launch configuration, so kernels may leave performance on the table. It proposes Proteus, a lightweight and portable JIT framework that extracts language-agnostic LLVM IR and specializes GPU kernels through a minimally intrusive annotation interface. The abstract claims end-to-end speedups over AOT compilation on both AMD and NVIDIA GPUs, and also reports better average performance than CUDA-specific Jitify, attributing gains to lower compilation overhead and sometimes faster generated binaries. If those results hold, the work is relevant as a compiler-runtime co-design approach for cross-vendor kernel optimization rather than a CUDA-only trick. The main missing detail is what specialization decisions are actually made, how large the runtime overhead is across workloads, and how fair the baselines are for each platform.

If implemented robustly, this offers a practical path to cross-vendor runtime kernel tuning that could improve performance without forcing applications into a CUDA-specific toolchain. A key caveat is that the abstract does not explain the specialization mechanism in enough detail to judge generality or reproducibility. Another caveat is that reported gains may depend heavily on workload selection, caching behavior, or baseline configuration fairness across vendors and toolchains, which is not clear from the abstract alone.

### 6. Tackling the Matrix Multiplication Micro-Kernel Generation with Exo.

*CGO | 2024 | Adrián Castelló 0001, Julian Bellavita, Grace Dinh, Yuka Ikarashi et al.*

> This paper shows how Exo can be used to generate GEMM micro-kernels that aim to match or exceed hand-written intrinsic or assembly implementations while reducing per-architecture implementation effort.

This paper targets a long-standing systems problem: high-performance GEMM depends on architecture-specific micro-kernels, but writing them by hand is expensive and hard to port. The authors present a step-by-step workflow in Exo for generating these micro-kernels, with the hardware target described through a concise instruction-library specification rather than a fully manual low-level rewrite. The abstract claims the resulting kernels perform close to, and sometimes better than, manually developed kernels written with intrinsics or assembly. If this holds, the main value is compiler-assisted production of near-expert GEMM kernels with a cleaner portability story. The biggest missing detail is evaluation scope: the abstract does not make clear which architectures, baselines, kernel shapes, or tuning effort were used to support the comparison.

For compiler and kernel-generation work, the payoff is a more maintainable path to high-performance GEMM kernels that could reduce retargeting effort across architectures. A key caveat is that the abstract does not say whether the gains hold across many architectures or only a narrow set of targets. Another caveat is that performance claims may depend on substantial manual guidance inside the Exo workflow, which the abstract does not quantify clearly as automation versus expert effort.
