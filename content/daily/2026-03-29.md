### 1. Q-Hitter: A Better Token Oracle for Efficient LLM Inference via Sparse-Quantized KV Cache.

*MLSys | 2024 | Zhenyu Zhang 0015, Shiwei Liu 0003, Runjin Chen, Bhavya Kailkhura et al.*

> The paper argues that KV-cache sparsification and low-bit quantization should be co-designed, and proposes a token selection rule that favors both attention importance and quantization friendliness during LLM decoding.

This paper targets a core LLM inference bottleneck: long-context KV cache storage and bandwidth during autoregressive decoding. It argues that combining cache sparsification with aggressive low-bit quantization is not straightforward, because quantization can distort heavy-hitter token selection and some selected tokens are inherently poor candidates for low-bit compression. The proposed Q-Hitter rule adds a quantization-friendliness signal on top of accumulated attention scores so token retention is chosen with both importance and compressibility in mind, apparently with layer-aware behavior. The abstract supports the claim that naive post-sparsification quantization is suboptimal, especially at 4-bit and below, but it does not make the evaluation setup, hardware assumptions, or exact empirical gains clear. If the method holds up, it is relevant as a practical KV-cache reduction strategy for bandwidth-limited LLM serving, but the missing implementation and baseline details matter for judging real systems value.

If the method is robust, it could improve KV-cache compression efficiency for LLM serving by cutting memory and bandwidth cost while preserving generation quality better than decoupled sparsification and quantization. A key caveat is that the abstract is truncated and underspecified, so the mechanism and evaluation details are incomplete. Another caveat is that the proposed rule appears heuristic, so generalization across models, prompts, and hardware is still unclear from the abstract alone.

### 2. From Threads to Tiles: T2T, a Compiler for CUDA-to-NPU Translation via 2D Vectorization.

*CGO | 2026 | Shuaijiang Li, Jiacheng Zhao, Ying Liu 0055, Shuoming Zhang et al.*

> This paper presents a compiler that translates CUDA kernels to tile-oriented NPUs by turning implicit SIMT thread parallelism into explicit 2-D vectorized loop structure.

This paper targets a real compiler gap between CUDA s fine-grained thread model and NPUs that execute explicit 2-D tile operations. It proposes T2T, which first rewrites CUDA kernels into an analyzable loop-nest form through a Unified Parallelism Abstraction and then performs 2-D vectorization by choosing loop pairs to map onto NPU tile instructions. The abstract claims this design also handles performance-sensitive CUDA semantics such as control flow and warp intrinsics through semantics-preserving optimizations. Reported evidence is reasonably concrete at the headline level: an implementation on Polygeist, evaluation on representative NPU architectures, up to 6.9x over baseline translation methods, and up to 73% of native CUDA performance on an A100 reference point. The main missing details are baseline fairness, which kernels and NPUs were used, and how much of the gain comes from the vectorization strategy versus backend-specific lowering and optimization.

If the method holds up, it offers a plausible compiler path for reusing CUDA kernels on non-GPU accelerators and clarifies where tile-oriented hardware can recover performance from thread-oriented source programs. A key caveat is that the abstract does not explain whether the reported performance holds broadly or is concentrated in kernels that already map cleanly to 2-D tiles. Another caveat is that using native CUDA on A100 as a reference point may be informative but does not by itself establish practical competitiveness on deployed NPUs without hardware-normalized context.

### 3. LLM-VeriOpt: Verification-Guided Reinforcement Learning for LLM-Based Compiler Optimization.

*CGO | 2026 | Xiangxin Fang, Jiaqin Kang, Rodrigo Rocha, Sam Ainsworth 0001 et al.*

> This paper trains a small LLM for LLVM IR peephole optimization by using formal-verification feedback as an RL reward, aiming to make generated rewrites both correct and useful.

The paper targets a core failure mode in LLM-based compiler optimization: generated low-level rewrites are often invalid, which blocks practical use even when the model can suggest interesting transformations. It proposes LLM-VeriOpt, which trains Qwen-3B with GRPO and uses Alive2 semantic-equivalence results inside the reward so the model learns to prefer verifiably correct LLVM IR edits. The abstract reports materially better rewrite success and correctness than prompt-only Qwen-3B and stronger results than larger prior LLM baselines, while also claiming speedups comparable to LLVM's handwritten `instcombine` on its peephole-optimization setting. For systems readers, the main value is the verifier-in-the-loop training recipe rather than generic code generation, because the gain is framed as coming from correctness-aware optimization search. The biggest missing detail from the abstract is evaluation setup and fairness: it does not make clear the benchmark mix, hardware/compiler conditions, training cost, or how broad the claimed optimization coverage really is.

If the method holds up, it offers a practical way to couple LLM optimization search with formal correctness checks, which is directly relevant to compiler and low-level code-generation workflows. A key caveat is that the abstract only supports a narrow peephole-optimization setting, so generalization to broader compiler passes is unclear. Another caveat is that baseline fairness is not fully assessable from the abstract, especially against larger models and handwritten LLVM passes under matched conditions.

### 4. Oaken: Fast and Efficient LLM Serving with Online-Offline Hybrid KV Cache Quantization.

*ISCA | 2025 | Minsu Kim 0004, Seongmin Hong, Ryeowook Ko, Soongyu Choi et al.*

> This paper proposes a hybrid online-offline KV-cache quantization scheme plus hardware support to reduce memory-bandwidth and capacity pressure in batched long-context LLM serving.

This paper targets a real serving bottleneck: batched attention in LLM inference is heavily constrained by KV-cache bandwidth and capacity, especially as context lengths grow. The core idea is to avoid expensive online outlier detection by setting outlier thresholds offline and then using those thresholds for lightweight online quantization-scale decisions. The paper also argues that algorithmic gains require hardware support, so Oaken includes custom quantization/dequantization engines and memory-management units intended for integration with LLM accelerators. If the design works as claimed, it could improve serving efficiency without paying the usual accuracy or online quantization overhead penalty. The biggest missing detail from the provided abstract is the actual evaluation evidence, hardware assumptions, and how much accuracy-performance tradeoff is achieved in practice.

If the claimed tradeoff holds, the work could make LLM serving less dependent on high-bandwidth, low-capacity memory by shrinking KV-cache cost at the algorithm-hardware boundary. A key caveat is that the abstract is truncated, so the strength of the empirical evidence cannot be judged from the provided text. Another caveat is that offline thresholds may be brittle if activation or KV statistics shift across models or workloads, which could hurt accuracy or require retuning at deployment time.

### 5. Proteus: Portable Runtime Optimization of GPU Kernel Execution with Just-in-Time Compilation.

*CGO | 2025 | Giorgis Georgakoudis, Konstantinos Parasyris, David Beckingsale*

> This paper presents a portable JIT path that specializes GPU kernels from LLVM IR at runtime, aiming to recover optimization opportunities that ahead-of-time compilation misses.

The paper targets a familiar systems problem in GPU HPC: ahead-of-time compilation cannot use runtime information such as actual inputs and launch settings, so kernels may leave performance on the table. Proteus addresses this by extracting language-agnostic LLVM IR and recompiling it just in time through a lightweight, annotation-driven interface that specializes kernels for runtime arguments and launch parameters. The abstract claims end-to-end speedups over AOT compilation on both AMD and NVIDIA GPUs, and also reports an average advantage over CUDA-specific Jitify, attributing gains to lower compilation overhead and sometimes better generated binaries. For readers interested in compiler/runtime co-design, the main attraction is the portability claim across vendors rather than a CUDA-only tuning story. The biggest missing detail is how much of the gain comes from specialization versus JIT overhead tradeoffs, and the abstract does not make the evaluation setup or baseline fairness fully clear.

If the claims hold, Proteus offers a practical route to cross-vendor GPU kernel specialization that could improve performance without committing to a CUDA-specific runtime toolchain. A key caveat is that the abstract does not separate compile-time overhead, cache effects, and steady-state kernel-speed improvements. Another caveat is that the reported wins may depend heavily on workload characteristics and may not generalize to short-running kernels or latency-sensitive settings.

### 6. Tackling the Matrix Multiplication Micro-Kernel Generation with Exo.

*CGO | 2024 | Adrián Castelló 0001, Julian Bellavita, Grace Dinh, Yuka Ikarashi et al.*

> This paper presents an Exo-based workflow for generating GEMM micro-kernels that aims to approach or exceed hand-written intrinsic or assembly implementations while improving portability across hardware targets.

This paper targets a long-standing systems problem: high-performance GEMM depends on hardware-specific micro-kernels that are costly to build and maintain by hand. It proposes a step-by-step method in Exo for generating these micro-kernels, with the hardware target described through a compact library-level instruction specification rather than fully manual low-level coding. The abstract claims the resulting kernels perform close to, and sometimes better than, manually developed intrinsic or assembly micro-kernels, which is the main evidence offered. If the claim holds, the value is a more portable and less labor-intensive path to micro-kernel development, but the abstract does not make the evaluation setup, target architectures, or fairness of the comparisons clear.

For compiler and systems work, the main payoff is a potentially more portable and maintainable path to near-hand-tuned GEMM kernels with less architecture-specific manual effort. A key caveat is that the abstract does not specify the experimental setup, so the strength and generality of the performance claim are hard to judge. Another caveat is that portability may mainly mean easier retargeting within Exo, not guaranteed strong performance across many architectures without additional tuning work.
