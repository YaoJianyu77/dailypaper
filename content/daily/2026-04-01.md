### 1. Improving Efficiency of GPU Kernel Optimization Agents using a Domain-Specific Language and Speed-of-Light Guidance

*arXiv preprint | 2026-03-30T21:16:39Z | Siva Kumar Sastry Hari, Vignesh Balaji, Sana Damani, Qijing Huang et al.*

> This paper argues that GPU kernel optimization agents become more efficient when they search in a compact CUTLASS-oriented DSL and stop or steer search with speed-of-light performance bounds instead of iterating blindly over low-level code.

The paper studies how to make LLM-based GPU kernel optimization agents spend fewer iterations on compile-profile-search loops across a large design space. Its main idea is to move the agent from low-level code generation to a compact DSL, CUTLASS, that preserves major kernel optimization choices while removing low-value detail, then add speed-of-light guidance to estimate headroom, steer trial selection, and budget search. The abstract reports results on 59 KernelBench problems showing that DSL-based search turns a regression into a speedup over PyTorch, and that adding SOL guidance improves speedup further while reducing token cost and helping weaker models beat stronger baseline agents on efficiency. Practically, this is interesting because it ties LLM agent quality to the search interface and to first-principles performance modeling rather than only to model size. The biggest detail still needing a full read is how fair and reproducible the evaluation is, especially the baseline agent setup, the SOL estimation procedure, and the criteria for detecting benchmark-gaming kernels.

If the method holds up, it suggests a practical recipe for cheaper and more reliable LLM kernel-tuning agents: constrain the action space at the right abstraction level and couple search to hardware-aware performance bounds. A key caveat is that the reported gains may depend strongly on the chosen benchmark suite, hardware target, and CUTLASS-backed search space. Another caveat is that a DSL can improve efficiency by narrowing search, but it may also hide optimizations available in lower-level code paths for some kernels or architectures.

### 2. Understand and Accelerate Memory Processing Pipeline for Disaggregated LLM Inference

*arXiv preprint | 2026-03-30T21:03:39Z | Zifan He, Rui Ma, Yizhou Sun, Jason Cong*

> This paper reframes long-context and retrieval-style LLM inference as a four-stage memory processing pipeline and argues that the irregular parts are a better fit for heterogeneous GPU-FPGA execution than for GPU-only serving.

The paper targets a systems bottleneck that becomes more important as LLM inference relies more on sparse attention, retrieval-augmented generation, and compressed memory for long-context reasoning. It proposes a unified four-step memory processing pipeline, then uses profiling to argue that memory-related work can consume a large share of end-to-end inference time and that its operators have heterogeneous compute behavior. Based on that view, the system offloads sparse, irregular, and memory-bound stages to FPGA while keeping compute-heavy stages on GPU in a disaggregated heterogeneous design. The abstract reports speedup and energy reductions against a GPU baseline on MI210 plus U55C, and says similar trends hold on A100, suggesting the gains come from matching operator type to hardware rather than from a single-model trick. The main missing details are how fair the baseline is, how data movement is handled, and which exact memory-processing subroutines dominate under different inference optimizations.

If the decomposition is robust, it gives a concrete systems template for accelerating retrieval- and memory-heavy LLM inference with hardware specialization instead of treating all serving work as GPU-native. A key caveat is that the abstract does not specify the fairness and strength of the GPU baseline, so the reported gains are hard to calibrate. Another caveat is that the deployment cost of adding FPGA and cross-device data movement may offset practical benefits in some serving settings despite positive benchmark results from the abstract alone.

### 3. Compiling Code LLMs into Lightweight Executables

*arXiv preprint | 2026-03-31T14:40:19Z | Jieke Shi, Junda He, Zhou Yang, Chengran Yang et al.*

> Ditto packages compressed Code LLMs and their inference path into standalone executables by combining parameter quantization with LLVM-level GEMV replacement, targeting faster local code generation on CPU-class devices.

This paper targets local deployment of Code LLMs on commodity machines, where memory limits and weak accelerators make standard inference pipelines expensive. It proposes Ditto, which combines product-quantization-like model compression with synthesized inference code, then adds an LLVM pass that swaps unoptimized GEMV calls for BLAS-backed implementations. The abstract claims this yields substantially faster inference and lower memory use on three Code LLMs while keeping pass@1 close to full precision. For systems readers, the practical angle is clear: it treats model compression and low-level executable generation as one deployment pipeline rather than separate steps. The main missing details are hardware assumptions, which model families and sequence settings were tested, and how fair the baseline inference pipeline really is.

If the claims hold, this could make offline Code LLM deployment more practical by reducing both runtime cost and packaging overhead through compiler-aware executable generation. A key caveat is that the abstract does not clarify whether the main gains come from compression, compiler rewrites, or an especially weak baseline. Another caveat is that results may depend heavily on CPU, BLAS library, and model architecture, which limits transferability until the setup is spelled out clearly.
