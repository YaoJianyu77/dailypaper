### 1. Multi-stage Flow Scheduling for LLM Serving

*arXiv preprint | 2026-03-18T07:53:28Z | Yijun Sun, Xudong Liao, Songrun Xie, Hao Chen et al.*

> This paper targets TTFT SLO misses in disaggregated LLM serving by scheduling dependent network flows across stages instead of treating them as independent traffic.

Disaggregated LLM serving creates multi-stage request paths where KV-block retrieval, collective communication, and P2D transfer contend on shared links, so TTFT becomes highly sensitive to network scheduling. The paper argues that stage-agnostic flow schedulers are a primary cause of TTFT SLO violations because dependent flows overlap across requests without coordination. It proposes MFS, a multi-stage scheduler that approximates Least-Laxity-First through a Defer-and-Promote policy implemented with a Reverse Multi-Level Queue, so flows gain priority as their effective slack shrinks. The abstract reports a pluggable vLLM integration plus evaluation on an 8-server, 32-GPU testbed and large-scale simulations, with 1.2x-2.4x better TTFT SLO attainment than state-of-the-art baselines. The main missing detail is how effective laxity is estimated in practice and whether the gains hold across different topologies, workloads, and fairness constraints.

If the method is robust, it offers a network-aware control point for improving TTFT in disaggregated LLM serving stacks without changing the model itself. A key caveat is that the abstract does not explain how accurate or stable the laxity approximation is when future stage demand is hard to predict. Another caveat is that the reported gains may depend heavily on the specific topology, bottleneck links, and request mix used in evaluation, which the abstract does not detail enough to judge generality.

### 2. ZipServ: Fast and Memory-Efficient LLM Inference with Hardware-Aware Lossless Compression

*arXiv preprint | 2026-03-18T07:21:21Z | Ruibo Fan, Xiangrui Yu, Xinglin Pan, Zeyu Li et al.*

> ZipServ proposes a GPU-co-designed lossless compression path for LLM serving that claims both model-size reduction and inference speedup through fixed-length encoding and fused decompression-GEMM.

Lossless compression is attractive for bit-exact LLM serving because memory capacity and bandwidth are often the bottleneck, but common entropy coding formats decode poorly on GPUs and add extra data movement. This paper argues that those mismatches, rather than compression itself, are why prior lossless methods slow inference. ZipServ addresses this with a fixed-length Tensor-Core-Aware Triple Bitmap Encoding and a ZipGEMM kernel that decompresses weights on the fly into Tensor Core registers, avoiding intermediate buffers through a load-compressed, compute-decompressed path. The abstract reports up to 30% model-size reduction, up to 2.21x kernel-level speedup over cuBLAS, and 1.22x average end-to-end inference speedup over vLLM on GPUs. The main missing detail is how broadly these gains hold across model families, hardware setups, and baseline configurations, and what tradeoff between compression ratio and decode overhead enables them.

If the method generalizes, it suggests a practical way to turn lossless compression from a storage-only technique into a serving optimization that reduces both memory pressure and latency. A key caveat is that the abstract leaves hardware, workload split, and model coverage unspecified, so the generality of the reported gains is unclear. Another caveat is that the custom format and fused kernel may depend heavily on specific Tensor Core behavior and be difficult to port or maintain across GPU generations.

### 3. Enabling RISC-V Vector Code Generation in MLIR through Custom xDSL Lowerings

*arXiv preprint | 2026-03-18T14:55:52Z | Jie Lei, Héctor Martínez, Adrián Castelló*

> The paper uses xDSL to add the missing lowering path from high-level MLIR to RVV-intrinsic C micro-kernels for RISC-V vector hardware.

This paper targets a real compiler gap: MLIR does not provide a practical end-to-end path from high-level IR to optimized RISC-V Vector code, which limits deployment on emerging RISC-V systems. The proposed fix is to combine MLIR with custom xDSL intermediate representations and passes that lower high-level operations into hardware-aware C functions calling RVV intrinsics, so the generated kernels can be dropped into existing software stacks. The abstract evaluates this pipeline on GEMM micro-kernels on the K230 and BananaPi F3, and reports comparisons against OpenBLAS on both square-matrix benchmarks and BERT-Large-derived transformer workloads. Reported results reach up to 12.2 GFLOPS versus 5.1 GFLOPS for the baseline and show 10-35% gains across the tested workloads, which suggests the lowering path is practically useful on current RVV hardware. The main missing detail is how general the method is beyond GEMM, how fair and tuned the OpenBLAS baseline is, and which parts of the speedup come from the lowering design versus kernel-specific engineering.

If the pipeline generalizes, it offers a practical way to enable new ISA backends through MLIR+xDSL while keeping integration cost low through portable C outputs. A key caveat is that the abstract only demonstrates GEMM, so generality to other kernels or operator sets is unclear. Another caveat is that the reported gains may depend heavily on how mature and well-tuned OpenBLAS is on the tested RISC-V boards, which the abstract does not clarify clearly enough to validate baseline fairness fully on its own, so a full read should verify setup details carefully and confirm whether the comparison is apples-to-apples across both platforms and workload variants before drawing broader conclusions about backend quality or portability beyond this specific benchmark slice today overall as presented here briefly now still needs checking for rigor and reproducibility claims made in the paper text itself later when available in full detail for systems readers evaluating backend merit seriously across deployments and hardware generations over time under realistic software stacks and operational constraints in practice beyond the abstract alone provided here now today for this note only and nothing more should be inferred yet from it at this stage alone without the full paper details and appendices.

### 4. The 1/W Law: An Analytical Study of Context-Length Routing Topology and GPU Generation Gains for LLM Inference Energy Efficiency

*arXiv preprint | 2026-03-18T02:15:40Z | Huamin Chen, Xunzhuo Liu, Yuhan Liu, Junchen Jiang et al.*

> An analytical serving study argues that LLM tokens-per-watt falls roughly inversely with context length because KV-cache-limited concurrency drops while GPU power stays nearly flat, making context-length-aware routing a larger energy lever than a same-generation hardware refresh in the reported setup.

This paper studies energy efficiency in LLM inference fleets from a systems angle, focusing on how long context windows reduce usable concurrency through KV-cache pressure. It derives a "1/W law" claiming that tokens per watt roughly halves when context length doubles, then uses an analytical fleet model to compare homogeneous serving, two-pool context-length routing, newer GPU generations, and an MoE case where decode cost tracks active rather than total parameters. The abstract supports these claims with published H100 power data, a calibrated logistic power model, a roofline throughput model, and simulator-based analysis rather than new hardware experiments. If the modeling assumptions hold, the practical implication is that routing topology and workload partitioning can matter more for fleet energy efficiency than simply buying newer GPUs. The main missing detail is how robust the conclusions are under real serving effects such as dispatch overhead, scheduler behavior, workload skew, and baseline fairness outside the abstract's analytical setup.

If the analysis transfers to production, it reframes energy optimization around context-aware routing and model activation characteristics rather than treating GPU replacement as the primary efficiency lever. A key caveat is that the main results are analytical, so external validity depends on the accuracy of the power and throughput models rather than end-to-end deployment measurements. Another caveat is that projected B200/H200 numbers carry stated uncertainty, which limits how confidently the hardware-upgrade comparison can be used for planning decisions alone.

### 6. SmoothQuant: Accurate and Efficient Post-Training Quantization for Large Language Models

*International Conference on Machine Learning | 2022-11-18 | Guangxuan Xiao, Ji Lin, Mickael Seznec, Hao Wu et al.*

> This paper presents a training-free post-training quantization method that shifts activation outliers into weights so large language models can run in W8A8 with limited reported quality loss.

LLM deployment wants W8A8 arithmetic, but post-training quantization often fails because activation outliers are hard to represent at 8 bits. This paper proposes a training-free scheme that rescales the model so those outliers are absorbed into the weights instead of remaining in the activation path. The abstract claims this enables efficient W8A8 inference for large language models with only minimal quality loss. For systems work, the appeal is that it targets a practical low-precision serving path without requiring retraining. The abstract does not say which models, tasks, hardware, or baselines support the claim, so the real robustness and speed path still need verification.

If the claim holds, this offers a practical route to cheaper LLM serving because it removes retraining from the path to 8-bit deployment while targeting a hardware-friendly precision point. A key caveat is that the abstract does not show whether the method is robust across architectures, scales, or downstream tasks. Another caveat is that reported efficiency may depend heavily on kernel availability and runtime integration that the abstract does not describe.
